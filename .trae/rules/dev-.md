# 开发规则
1.新增功能：优先新增文件 / 新增模块 / 新增函数，禁止在原有业务代码中硬塞逻辑，避免污染旧代码；
2.需求迭代 / BUG 修复时，严格遵循「最小化改造原则」，非必要不修改原有稳定代码，不影响原有功能逻辑；
3.函数只做单一逻辑，文件只承载单一模块能力；
4.单文件代码行数 ≤ 500 行，超量立即拆分模块 / 文件 / 子组件；
5.拆分优先级：先拆公共逻辑→再拆子组件 / 子模块→最后拆函数，拆分后保证模块间低耦合，无冗余依赖。
6.代码编写 / 迭代完成后，强制自查文件行数，不符合规范的即时调整；
7.BUG 修复：只修改触发 BUG 的精准代码段，禁止顺带做「代码优化、逻辑重构」；
8.禁止大面积删改代码，除非确认该代码无任何业务依赖；
9.注释仅标注核心业务逻辑、特殊处理、边界条件、非直观逻辑，杜绝冗余注释、逐行注释、无意义注释，减少无效代码量占用；
